# Pivots Ledger

The single source of truth for **load-bearing rules** and **carried-forward
deferrals** that no module README owns end-to-end. Cross-references the
authority documents (each row links back to the source-of-truth README,
commit, or spec section).

This file exists so the Phase 4 work cannot quietly miss a rule that was
locked in during Phase 2. Update this ledger whenever:

- A new hard rule is added to AGENTS.md
- A new deferral is named in a module commit/README
- A deferral is closed (mark with strike-through + closing commit hash)

Last reconciled: 2026-05-19. Reconciliation method: read every module
README + commit message from `0d4b95e` forward; rounding rule confirmed
present in `src/lib/recompute/README.md` + `src/lib/recompute/cents.ts`.

### Localization + plain-language addendum (added 2026-05-20, P4-prep)

| #    | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         | Source                                                             |
| ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------ |
| LOC1 | **US conventions + plain language in every user-facing string.** V1 is a US-only product. Locale toggles, country selectors, and i18n scaffolding are explicitly NOT in scope; adding them is a separate planned decision, not a casual addition. **US conventions:** Sunday–Saturday week (the dashboard's "Team This Week" widget uses Sunday-start); dates render `MM/DD/YYYY` ("05/20/2026"); currency renders `$X,XXX.XX` with US thousands separators (comma) and decimal (period); phone numbers render in US format (`(555) 123-4567` or `+1 555 123 4567`); addresses use a 2-letter state code + 5-or-9-digit ZIP; no country selector on address inputs; imperial units (miles, feet, fahrenheit) where units appear. **Plain language:** no developer jargon in user-facing strings (no "row," "record," "entity," "tuple," "primary key," "RLS," "validation error," "unauthorized" — say "Contact," "Event," "Task," "We can't show this to you," "Try again"). Button labels are verbs ("Save", "Create event", "Send invite" — not "Submit", not "OK"). Empty states say what to do next ("No contacts yet. Add your first contact." — not "No data."). Loading states have text ("Loading your events…" — not just a spinner). The role NAME `user` (per P4-roles) is enum-only; UI copy that names the people says "team" / "team member." **Implementation surfaces (V1):** `formatCents` + `formatDate` live at `src/lib/format/index.ts`. The dashboard's "Team This Week" widget hard-codes Sunday-Saturday. Phone/address/imperial-unit formatters land alongside the first surface that needs them; document them here when they do. | `src/lib/format/index.ts`; `src/modules/rbac/types.ts` (LOC1 note) |

### URL handling standard (added 2026-05-21, P4.2 micro-commit)

| #    | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               | Source                                                              |
| ---- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| URL1 | **Forgiving URL input across the CRM.** Every user-facing URL form field is a plain text input (no `type="url"`, no browser-side strict URL validation). On submit, `normalizeUrl()` from `@/lib/format/url.ts` auto-prepends `https://` when the user types `www.example.com` or `example.com`. Inputs that already start with a scheme (`http://`, `https://`, `mailto:`, `tel:`, `sms:`, `ftp:`) are passed through unchanged. Server-side Zod `z.url()` schemas stay strict — the form layer is responsible for handing them a normalized value. Applies to all current and future URL fields. | `src/lib/format/url.ts` + `V1_ROADMAP.md` "URL Validation Standard" |

### AI layer guiding principle (added with module 16a)

| #   | Rule                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         | Source                                                                                                   |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| AI1 | **The AI is a tool the human drives, never an autonomous actor.** It never self-directs, never decides on its own to build/modify/execute things it wasn't asked for, never enables its own workflows, never acts against client data or money without the human having explicitly asked for that specific action. Every AI write = a human request routed through the IDENTICAL human action path (same orgAction, same RLS, same validation, same audit). NO AI-specific back-channel. If a planned capability would (a) bypass orgAction/RLS/audit, (b) self-initiate an unrequested action, or (c) enable/fire a workflow without explicit human enable — STOP and flag. | `src/modules/ai-workflow-builder/README.md` §"AI LAYER GUIDING PRINCIPLE — IT IS A TOOL, NOT THE LEADER" |

## How to use this ledger

- **Before starting a Phase 4 module**, scan Section 2 for entries whose owner is your module or phase. Implement them or carry them forward with an updated owner.
- **Before reviewing a PR**, check Section 1 for the rules that apply to the changed code (most PRs touch 2-4 rules).
- **When closing a deferral**, move it from Section 2 to Section 3 with the closing commit hash + add the source-of-truth README link.
- **When adding a new hard rule** (rare), add it to AGENTS.md FIRST, then add a row to Section 1 here.

**Do NOT** treat this ledger as a substitute for the source READMEs. It is a navigation aid — every row points back to the authoritative document. The ledger can become stale; the READMEs are the source of truth.
